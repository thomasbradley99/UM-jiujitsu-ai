The mental model

We treat MuBit as three small services we glue together, not a framework we live inside. MuBit owns the prompt as a versioned object (like git for prompts) and the per-event outcome log. Every spin asks MuBit "what's the active version?", runs the pipeline locally, scores against ground truth locally, then tells MuBit how each event went. Periodically, MuBit's optimizer LLM reads our outcomes and rewrites the prompt.

The full SDK surface area

Every MuBit call lives in flywheel/mubit_client.py. Eight wrapper functions, five distinct SDK calls. Nothing else in flywheel/ imports mubit.

The four touch-points

1setup · provision the agent once

find_agent · create_agent_definition

flywheel/setup.py runs once per project. It calls find_agent(); if the agent doesn't exist in our MuBit project, it creates one with a precise role + description + the seed prompt from flywheel/prompts/verifier_v1.md. 

# flywheel/setup.py
existing = find_agent()                     # MuBit: list_agent_definitions
if existing: return                       # idempotent
create_agent(                              # MuBit: create_agent_definition
    role="bjj submission detector",
    description=_AGENT_DESCRIPTION,         # spells out exactly which prompt slab MuBit owns
    system_prompt=SEED_PROMPT.read_text(),  # verifier_v1.md → first version on MuBit
)

Key choice: the description we give MuBit is precise — it tells the optimizer that MuBit only owns the DOMAIN RULES layer, that framing/fighter-block/JSON-schema are locked, and that outcomes come from a windowed scan. The optimizer reads this every time it rewrites; it shapes how it interprets our rationales.

2predict · pull the active prompt at the start of every spin

get_prompt

flywheel/spin.py:predict() is one MuBit call. The version_id becomes the cache key for everything that follows — the prompt text we wrote to disk, the result.json we'll generate, the report we'll score. 

# flywheel/spin.py — predict()
prompt_text, version_id = get_active_prompt()      # MuBit: get_prompt
out_dir = OUTPUTS_DIR / "runs" / run_id / version_id
(out_dir / "domain_rules.md").write_text(prompt_text)  # local audit trail
_run_analyze(video, out_dir, rules_path, gt_path)    # subprocess into VLM-gemini/analyze.py

The pipeline runs as a subprocess because analyze.py uses google.generativeai while the MuBit SDK uses google-genai — one venv would conflict, two interpreters keep them clean. The active prompt gets injected into Stage 1 of analyze.py via its --domain-rules-file flag.

What the prompt looks like · only the middle slab changes between iterations:

FRAMING + FIGHTER BLOCKPython · locked
DOMAIN RULESMuBit · versioned
OUTPUT SCHEMA (JSON)Python · locked

This is the smallest unit of optimization that still meaningfully changes behaviour. Smaller and the optimizer can't move the metrics; larger and you can't isolate cause/effect.

3feedback · per-event outcomes with specific rationales

archive · record_outcome

This is the most important integration choice and the thing that actually makes the optimizer work. After measure() produces a Report with per-event details, we iterate the events and call two MuBit APIs per event. 

# flywheel/spin.py — feedback() (per event)
ref_id = archive_event(                          # MuBit: archive (the artifact)
    run_id, content=detailed_artifact_text,
    kind="pipeline_cluster_tp",                  # or _fp, _window_fn
    labels=["submission", "true_positive", "armbar"],
)
record_outcome(                                  # MuBit: record_outcome (the verdict)
    run_id, ref_id,
    success=True,                                # or False
    signal=0.85,                                # numeric reward in [-1, 1]
    rationale=specific_english_text,            # the optimizer reads THIS
)

Rationales are free-form English — the optimizer's LLM reads them as-is. Effort goes into making them concrete and actionable. Three styles by event status:

TP cluster Correctly detected armbar at 02:34 (within 3s of GT). The DOMAIN RULES produced YES on the windows around this moment. Keep the parts of the rules that produced this detection.
If technique was wrong, we add: "Mis-classified as guillotine; correct canonical name is armbar. Rules can be more specific…"
FP cluster (hallucination) Hallucinated submission at 03:55 (guillotine). The DOMAIN RULES were too liberal. Sample window reasoning: '<the actual model output for that window>'. Tighten the rules to require stronger corroboration…
FN (missed GT) — the killer feedback Missed real armbar by BALD at 01:08. 2 windows covered this moment and ALL returned is_submission=false. One window's reasoning: '<the actual NO reasoning>'. To recover this case, the rules likely need to allow end-of-round reset signals…
The FN rationale literally quotes the pipeline's own NO-reasoning back at the optimizer. The optimizer sees: "you said NO at the moment a real submission happened, here's why you said NO." That's how a strict-tap-only seed evolved into the reset-aware v4 prompt.

We also push a numeric signal alongside the rationale: +0.5..+1.0 for TPs scaled by technique/submitter correctness, -0.7 for FPs, -0.85 for FNs. MuBit can use this as a secondary input if it wants.

4improve · ask MuBit's optimizer for a new prompt

optimize_prompt · activate_prompt_version

One MuBit call plus optional auto-activation. request_optimization() returns a candidate prompt in pending state by default — a human approves in the MuBit Console. With auto_activate=True (used by flywheel.cli loop) we promote it ourselves so the next spin sees it as active. 

# flywheel/improve.py
resp = request_optimization()              # MuBit: optimize_prompt (LLM rewrite)
candidate  = resp["candidate"]              # {"version_id": "pv-377be9c6-…", ...}
summary    = resp["optimization_summary"]   # the optimizer's English rationale
confidence = resp["confidence"]             # 0.0–1.0

if auto_activate:
    activate_version(candidate["version_id"])  # MuBit: activate_prompt_version

The next spin's get_active_prompt() automatically sees the new version. Loop closed.

How outcomes group · session keys

Every outcome carries a session_id (we call it run_id in our wrapper). One run_id per video. Re-running the same video accumulates outcomes in the same MuBit session, so the optimizer sees the full history. Switching videos gets a new session, so outcomes don't bleed across datasets. 

def run_id_for_video(video_path: Path) -> str:
    return f"verify:{video_path.stem}"     # e.g. "verify:video"

Combined with agent_id, this gives MuBit enough structure: agent = which prompt-slab is being optimised, session = which dataset/run-context.

What MuBit is · what MuBit isn't

What MuBit does for us

  • Versioned prompt store with monotonic IDs — every result.json lives next to a domain_rules.md and a pv-… version_id you can look up in the Console.
  • Persistent outcome log keyed by version + session. We don't need our own database.
  • LLM rewriter that reads English rationales (not metrics). Quality of the rewrite scales with quality of the rationales we send.
  • Atomic publish. activate_prompt_version flips the active pointer; the next get_active_prompt sees it. This is what makes the autonomous loop possible.

What MuBit doesn't do (we do)

  • MuBit doesn't run inference. Gemini does that.
  • MuBit doesn't compute F1 or compare to ground truth. VLM-gemini/eval/ does that.
  • MuBit's optimizer doesn't see metrics. It sees our rationales. Generic feedback → generic rewrites.
  • MuBit doesn't prevent regressions. We saw v3 drop from 77% to 50% — the rewriter "improved" the prompt by its own logic but lost recall. Multi-iteration arcs are how we surface this.
