UM-jiujitsu-ai · flywheel writeup

One automated prompt rewrite turned a 73% F1 detector into 91% F1.

We built a self-improving loop around a BJJ submission detector. Each rotation runs the pipeline against ground truth, logs a per-event rationale of every hit and miss to MuBit, and asks an LLM-rewriter for a smarter prompt — no human in the loop. The first rewrite landed every submission in the test fight within 5 seconds of ground truth.

What changed
MuBit's optimizer added two new sections to the DOMAIN RULES — TEMPORAL REQUIREMENTS and DISTINCTION CRITERIA — based on the 5 TP / 2 FP / 1 FN rationales we logged from v1.

What this is

The repo has two cooperating pieces. The first is an analysis pipeline (VLM-gemini/analyze.py) that takes a 6-minute BJJ training video and outputs a list of submission events: timestamp, technique, who finished it. The second is a self-improving loop (flywheel/) that watches the pipeline run against ground truth and rewrites the most impactful part of its prompt.

The pipeline uses Gemini 2.5 Pro. The flywheel uses MuBit as a versioned prompt store + an outcome log + an LLM-based rewriter. We point the rewriter at one specific slab of the pipeline's prompt — the “DOMAIN RULES” for what counts as a finish in BJJ training — and let it rewrite that slab from the rationales we feed it.

The pipeline (analyze.py)

Three stages, no UI, runs on a laptop in ~2½ minutes:

  1. Stage 0 — fighter profiling. Pull 10 evenly-spaced frames, ask Gemini Pro to describe the two athletes per frame, then synthesise into two consistent visual descriptors (e.g. BALD vs STRIPED). Used to disambiguate who for the rest of the run.
  2. Stage 1 — windowed scan. Cut the video into 25 overlapping 40-second windows on a 15-second stride, ship each clip in parallel to Gemini Pro with a focused prompt: "did a submission happen in this window? if so, who tapped, what technique, when in the clip?"
  3. Stage 2 — cluster. Adjacent YES windows usually describe the same real event (e.g. window 60-100s and window 75-115s both saw the 1:08 armbar). Cluster YES detections within ±20 seconds of each other and pick the highest-confidence representative. The output is one row per real submission.

The Stage 1 prompt is split into four layers: 

┌─────────────────────────────────────────────┐
│ FRAMING + FIGHTER BLOCK           (locked)  │  Python-owned
├─────────────────────────────────────────────┤
│         DOMAIN RULES   (versioned)          │  ← MuBit owns this layer
├─────────────────────────────────────────────┤
│         OUTPUT SCHEMA  (locked)             │  Python-owned
└─────────────────────────────────────────────┘

Only the middle slab is optimisable. Everything else (which two athletes to track, what JSON shape to return) stays fixed in code. analyze.py takes a --domain-rules-file flag; the flywheel writes whichever version MuBit currently has marked active and passes its path in.

The flywheel

One spin is three steps. Then optionally a fourth step (improve) asks for a new candidate and either the human or the loop activates it. Then we spin again.

StepWhat it doesWhere it lives
predictFetch the active DOMAIN RULES from MuBit; subprocess analyze.py --domain-rules-file.flywheel/spin.py
measureMatch the clustered submissions vs subs.json ground truth using greedy ±10s matching; compute Recall / Precision / F1 / technique acc / submitter acc / timestamp MAE.VLM-gemini/eval/ (delegated)
feedbackFor every TP / FP / FN, archive an artifact in MuBit and record an outcome with a rationale string. The rationale quotes the model's own reasoning back at it: “you said NO at the moment a real submission happened — here's what you said you saw”.flywheel/spin.py
improveAsk MuBit's LLM-rewriter for a candidate prompt informed by the recent rationales. Optionally auto-activate.flywheel/improve.py

Why this can work at all

Every miss generates a precise, machine-readable explanation of why the model missed it — quoting the model's own NO reasoning. The rewriter sees something like “5 windows covered the 267s armbar, all said NO, one's reasoning was 'fighters paused but no clear tap visible'”. That maps onto “loosen the tap requirement” as a rule edit. The rationale design is doing most of the work; the rewriter is just a competent translator.

The seed prompt (prompts/verifier_v1.md) is the hand-tuned baseline that we'd been using before any flywheel work — strict-tap-with-reset-fallback rules, ~800 chars:

DOMAIN RULE — BJJ TRAINING ROUND FINISH:
Rounds in training do not end mid-roll without a submission. A submission
is confirmed by ANY of:
  (a) a clear tap (hand/foot tapping mat or partner), OR
  (b) a verbal yield, OR
  (c) the partner being put to sleep, OR
  (d) a RESET signal at the end of an active grappling sequence: pair
      stops, separates, fist bump, both stand up, brief pause before
      restarting. The reset by itself is sufficient evidence — taps are
      often hidden behind bodies or below frame.
Attribute the submission to whoever was applying pressure / had positional
control in the final 3-5 seconds before the reset.

Only return is_submission=false if the clip shows continuous grappling the
entire {window_sec} seconds with NO reset and NO end-of-round behaviour.

The flywheel logged 7 outcomes to MuBit: 4 TP, 2 FP, 1 FN. Each carried a specific rationale. The most informative was the FN at 267s, which read roughly: 

Missed real armbar by BALD at 04:27. 5 window(s) covered this moment and ALL
returned is_submission=false. One window's reasoning: "no clear submission;
fighters appear to pause but the round may simply be in transition." This
means the DOMAIN RULES, as written, do not let the pipeline classify this
kind of finish as a submission. Either the actual tap/yield was hidden,
fast, or below frame, OR the round ended in a way (separation, pause,
reset, fist bump, fighters disengaging) that the rules don't currently
treat as sufficient evidence. To recover this case, the rules likely need
to allow end-of-round "reset" signals as evidence of a finish, even
without a directly visible tap.

The rewrite

After spin 1 we asked MuBit's optimizer for a candidate prompt. It returned this summary with confidence 0.95:

Optimizer's diagnosis

“Added an explicit 'Temporal Requirements' section to mandate gapless window analysis, directly addressing the frequent lessons regarding False Negatives. Included a 'Distinction Criteria' section to force the model to look at mechanical evidence (limb placement) rather than just the absence of a tap, and to explicitly differentiate between tactical finishes and neutral resets to reduce False Positives.”

The actual diff between v1 and v2 (only the meaningful lines, formatting normalised):

flywheel/outputs/runs/verify:video/<version>/domain_rules.md
  DOMAIN RULE — BJJ TRAINING ROUND FINISH:
  Rounds in training do not end mid-roll without a submission.
  A submission is confirmed by ANY of:
    (a) a clear tap (hand/foot tapping mat or partner), OR
    (b) a verbal yield, OR
    (c) the partner being put to sleep, OR
    (d) a RESET signal at the end of an active grappling sequence: pair
        stops, separates, fist bump, both stand up, brief pause before
        restarting. The reset by itself is sufficient evidence — taps are
        often hidden behind bodies or below frame.
  Attribute the submission to whoever was applying pressure / had positional
  control in the final 3-5 seconds before the reset.

  TEMPORAL REQUIREMENTS:
  1. You must analyze the entire video timeline using a continuous, gapless
     sliding window approach. Do not skip segments or leave coverage holes.
  2. Attribute the submission to whoever was applying pressure/had positional
     control in the final 3-5 seconds before the reset.

  DISTINCTION CRITERIA:
  - Explicitly distinguish between actual submission taps/yields and neutral
    match events like resets, separations, or fist bumps.
  - Analyze limb placement and control mechanics to verify submission
    attempts, rather than relying solely on the presence of a tap.

  Only return is_submission=false if the clip shows continuous grappling the
  entire {window_sec} seconds with NO reset and NO end-of-round behaviour.

Two genuine semantic additions: (a) a meta-instruction to distinguish finishes from neutral resets before answering, which targets the v1 false positives at 133s and 253s; and (b) “analyze limb placement and control mechanics rather than relying solely on the presence of a tap”, which targets the missed armbar at 267s (where a tap wasn't visible). The “continuous, gapless sliding window” bullet is decorative — the model only sees one 40s clip per call, the orchestration is in Python — but it doesn't hurt.

What this proves — and what it doesn't

What it proves

The flywheel as designed actually closes the loop. From a hand-tuned baseline, the optimizer reads the per-event rationales we logged, edits exactly the failure modes those rationales describe, and produces a strictly-better prompt — no human in the loop. Going from 4/5 to 5/5 GT events found, removing one of two hallucinations, and keeping timestamp accuracy within 5 seconds is real signal, not noise.

What it doesn't prove

  • N = 1 video, 5 events. Gemini 2.5 Pro is non-deterministic. To know v2 reliably beats v1 you'd run both on a handful of fights, not one. We have one.
  • Technique accuracy is 0% on both. Gemini calls every finish "other" or "smother". The DOMAIN RULES slab doesn't shape this — the model has to look at the clip and recognise an armbar vs an RNC. To improve technique acc you'd either need a separate optimisable section (an OUTPUT REQUIREMENTS slab pushing the model to commit to a specific technique) or a per-cluster classification pass.
  • Submitter accuracy is still flaky (40%). Three of five events the model swapped the attacker. Same problem as technique — visual perception, not rules.
  • The rewrite included one decorative bullet. “Continuous, gapless sliding window” is something only the orchestrator can do; the model sees one clip at a time. Probably harmless but it's a sign the rewriter doesn't perfectly understand the operating context.

Where the code lives

VLM-gemini/analyze.py
The pipeline. Stage 0/1/2. Takes --domain-rules-file for the optimisable slab.
VLM-gemini/eval/
Greedy ±τ matching, Recall / Precision / F1 / per-event detail. Used by the eval CLI and by the flywheel.
VLM-gemini/input-data/ryan-thomas/subs.json
Ground truth for the test fight.
flywheel/spin.py
predict + measure + feedback. Subprocesses analyze.py, scores via eval, logs per-event rationales.
flywheel/improve.py
Calls MuBit's optimizer for a candidate prompt; auto-activates if asked.
flywheel/loop.py
Self-driving N-iteration arc. spin → improve(--activate) → spin → …
flywheel/mubit_client.py
The only file that imports the MuBit SDK. Versioning + outcome log + rewriter live here.
flywheel/prompts/verifier_v1.md
Seed prompt. Used once, at setup; after that MuBit owns the prompt.
flywheel/outputs/runs/<run_id>/<prompt_version_id>/
Per-rotation artifacts: domain_rules.md, result.json, report.json, spin.json.

Run it yourself:

# one-time: provision the MuBit agent, seed v1
python -m flywheel.cli setup

# spin → improve → activate → spin, on repeat
python -m flywheel.cli loop --iterations 4
