# Results — what the flywheel actually produced

> **For hackathon judges.** This page is the one-stop tour of every file
> behind our submission. Every number below is reproduced from a JSON
> committed in this repo.

## TL;DR

We treat the BJJ submission-detection prompt as a versioned, testable
artifact. MuBit stores prompt versions, logs per-event outcomes, and
proposes the next prompt. We score every version against ground-truth
submission timestamps with the existing `VLM-gemini/eval/` module. Then
we activate the new version and spin again. Same fight, smarter rules,
F1 climbs.

The headline arc — open this in a browser:

  **`flywheel/outputs/arc_report_handtuned.html`** *(side-by-side, prompt diffs, per-event)*

| Iter | Prompt version (MuBit ID) | F1 | Precision | Recall | Matched | Hallucinations |
|------|---------------------------|----|-----------|--------|---------|----------------|
| v1 (seed)            | `pv-ac5575b2-…` | 57% | 44% | 80%  | 4/5 | 5 |
| v2                   | `pv-b535d177-…` | 77% | 62% | 100% | 5/5 | 3 |
| v3 *(regression)*    | `pv-853f6c04-…` | 50% | 43% | 60%  | 3/5 | 4 |
| **v4 (perfect)**     | `pv-377be9c6-…` | **100%** | **100%** | **100%** | **5/5** | **0** |

Same 6-min fight, same pipeline, same model. The only thing that
changed is the DOMAIN RULES block of the scan prompt that MuBit owns.
Bonus: at v4 the timestamp MAE is **2.4s** and submitter attribution
accuracy is **80%** (4/5) — see `…/pv-377be9c6-…/spin.json`.

## Where every artifact lives

### 1. The headline visual reports

```
flywheel/outputs/
├── arc_report_handtuned.html     ← open this first  (the F1 0.57 → 1.00 climb)
└── arc_report_naive.html         ← contrast: naive auto-loop (F1 drifts)
```

Both are self-contained HTML — no server, just `open` them. They show:

  - per-iteration metrics (F1 / P / R / matched / hallucinations)
  - the **diff** of the prompt vs the previous version
  - the optimizer's free-text rationale for each rewrite
  - per-event outcomes (matched / missed-gt / hallucination) with
    timestamps, claimed technique, and Δt vs ground truth

Both arcs were generated by the same `flywheel.cli loop` command —
identical pipeline, identical model, identical optimizer. The only
thing that differs is the seed prompt and the MuBit agent used:

  - **naive arc** — agent `submission-detector-naive`, seed
    `flywheel/prompts/verifier_naive.md` (literally one sentence:
    "Find moments in this video where someone gets submitted").
    Optimizer adds structure but has no anchor, so it drifts.

  - **handtuned arc** — agent `submission-verifier`, seed
    `flywheel/prompts/verifier_v1.md` (a few rules grounded in the
    BJJ-training reality: tap / yield / unconscious / reset signals).
    Optimizer refines the rules, eventually hits 100% F1.

The contrast is the demo: an auto-rewriter with no ground-truth
feedback can drift backwards from a vague seed; the same rewriter on a
domain-grounded seed climbs to perfect.

### 4. Cross-eval — does it generalise?

We ran each of the 4 prompts on a **second video** (`chris-instructor`,
held out — different gym, different fighters, different camera
angle). Same script: `bash flywheel/scripts/cross_eval.sh chris-instructor`.

```
flywheel/outputs/cross_eval/chris-instructor/
├── v1-pv-ac5575b2/   F1 = 100%   (4/4 matched, 0 halluc)
├── v2-pv-b535d177/   F1 =  75%   (3/4 matched, 1 halluc)
├── v3-pv-853f6c04/   F1 =  29%   (1/4 matched, 2 halluc)
└── v4-pv-377be9c6/   F1 =  50%   (2/4 matched, 2 halluc)
```

Honest finding: **v4 overfit to the original fight**. On the held-out
video, v1 (the seed) generalises best; v2 is a reasonable middle.
This is exactly the failure mode you should expect from a rewriter
that only sees one training video, and it's why we ship the
cross-eval next to the headline. We talk about it in the README under
"What we learned".

### 5. Source code that produced everything

```
flywheel/
├── prompts/verifier_v1.md  ← the seed (v1) prompt that bootstraps the arc
├── mubit_client.py         ← the only file that imports the MuBit SDK
├── setup.py                ← create the agent + seed v1
├── spin.py                 ← predict (analyze.py) + measure + feedback
├── improve.py              ← ask MuBit's optimizer for a candidate
├── loop.py                 ← orchestrate spin → improve → activate → spin
├── report.py               ← render the side-by-side HTML you saw
├── cli.py                  ← CLI entry: setup / spin / improve / loop / report
└── scripts/cross_eval.sh   ← run the 4 prompts on a held-out video
```

The pipeline being optimised: `VLM-gemini/analyze.py` (Stages 0–2 of
the BJJ submission detector). The DOMAIN RULES block is the only
layer that changes between versions; everything else (windowing,
fighter framing, output schema, clustering) stays fixed.

## How to reproduce from scratch

```bash
source .venv/bin/activate

# === 1) the handtuned arc that hit 100% F1 ============================
# Use the verifier seed and a dedicated agent so it doesn't clobber the
# naive agent's history.
export FLYWHEEL_AGENT_ID=submission-verifier
export FLYWHEEL_SEED_PROMPT=verifier_v1.md

python -m flywheel.cli setup                  # idempotent: creates agent + seeds v1
python -m flywheel.cli loop --iterations 4    # 4 spins, ~12 min on the 6-min video

# Render the side-by-side HTML
python -m flywheel.cli report \
  --version-a pv-ac5575b2-da74-46ed-93c0-af646accf89b \
  --version-b pv-377be9c6-d965-4e43-9511-43ab10ecd725
# → flywheel/outputs/runs/verify:video/report.html

# === 2) the naive arc (drift) =========================================
unset FLYWHEEL_AGENT_ID FLYWHEEL_SEED_PROMPT     # falls back to default = naive
python -m flywheel.cli setup
python -m flywheel.cli loop --iterations 6

# === 3) cross-eval the 4 handtuned prompts on a held-out video ========
bash flywheel/scripts/cross_eval.sh chris-instructor
```

(The exact MuBit `pv-…` IDs you get back will be different — MuBit
allocates a fresh version_id for each new agent — but the F1 trajectory
should match within stochastic noise.)

Cost: ~3 min/spin on a 6-min video (Gemini 2.5 Pro / 3 Flash, 12-way
parallel windows). Cached per `prompt_version_id`, so a 4-iter loop is
~12 min wall-clock the first time and ~30s on a re-run.

Env required (in `.env.local` at repo root):

```
GEMINI_API_KEY=...
MUBIT_API_KEY=mbt_...
```

## What's actually optimised (the prompt slab)

The BJJ scan prompt is layered. MuBit owns one slab.

```
┌─────────────────────────────────────────────┐
│ FRAMING + FIGHTER BLOCK           (locked)  │  Python
├─────────────────────────────────────────────┤
│         DOMAIN RULES   (versioned)          │  ← MuBit slot
├─────────────────────────────────────────────┤
│         OUTPUT SCHEMA  (locked)             │  Python
└─────────────────────────────────────────────┘
```

The seed v1 is deliberately strict (tap / verbal yield / unconscious
only). Real submissions in training don't always meet that bar — taps
are fast, hidden, below frame. The optimizer's job is to learn from FN
windows that "round-end / reset / disengagement" is also evidence of a
finish, then trade off precision and recall as it goes.

The full v4 prompt that scored 100% F1 lives at:

  `flywheel/outputs/runs/verify:video/pv-377be9c6-d965-4e43-9511-43ab10ecd725/domain_rules.md`

    JPEGs of fighter-profile frames for Stage 0. Multi-MB per run, fully
    regenerable. **Gitignored.**
  - Scratch runs from earlier sweeps (`pv-1daf688b`, `pv-23a69edf`, etc.) —
    not part of the headline arc. **Gitignored.**
  - The source video `VLM-gemini/input-data/ryan-thomas/video.mov` — kept
    locally only (private footage, also too large). **Gitignored** by the
    repo-root `.mov` rule. Ground truth is committed
    (`VLM-gemini/input-data/ryan-thomas/subs.json`).
